# Фильтры задач и компактные меню GUI

## Кратко

В GUI добавлены компактные верхние меню и фильтр задач. Фильтр позволяет:

- показывать завершенные, архивные и отмененные задачи;
- ограничивать выборку задач диапазоном дат;
- применять фильтр без изменения текущей вкладки Google Sheets.

При включении завершенных задач GUI автоматически включает диапазон дат за последний месяц, чтобы по умолчанию не загружать слишком большой набор данных.

## Сценарий в GUI

Верхняя панель разделена на три меню:

- `Задачи`: загрузка задач, создание рейса, обновление;
- `Обработка`: обработка всех ТС, Wialon, Я.Карты;
- `Справочники`: история, пользователи, настройки.

Контекстное меню строки таблицы содержит быстрые действия для выбранного рейса:

- `Завершить рейс`;
- `История` — открывает диалог истории навигации для строки, по которой вызвано контекстное меню;
- `ПИН-коды`;
- `Добавить заметку`;
- управление подсветкой строки.

В диалоге истории навигации строки типа `заметка` подсвечиваются цветом `#fdc46f`. Заметка может содержать текст и несколько вложений фото/видео/файлов; в истории показываются превью всех доступных вложений, а не только первого.

Кнопка фильтра открывает небольшую панель с настройками:

- `Завершённые`;
- `Даты`;
- дата начала;
- дата окончания;
- `OK` / `Отмена`.

Горячая клавиша `F5` обновляет текущую таблицу.

## Редактор ID-справочника ТС

Диалог `Редактор ID-справочника` открывается для ручного управления привязками ТС к ID мониторинга.

Разметка диалога:

- сверху находится поиск по видимым полям таблицы;
- таблица показывает только рабочие поля `ИД-Объекта`, `Наименование`, `ТС`;
- служебное поле `Центр мониторинга` (`CENTER_FIELD`) не выводится в таблицу, но остаётся в данных и сохраняется для новых записей как `DEFAULT_MONITORING_CENTER`;
- строка добавления `ИД-Объекта` / `Наименование (пример: К532СМ750)` / `Добавить` расположена под таблицей;
- кнопка `Удалить` удаляет выбранную строку из локального списка, окончательная запись в БД выполняется только после `Сохранить`.

Удаление записи из справочника выполняется явно через `vehicle_repository.delete_registry_entry()`. Для API-клиента это уходит в `DELETE /api/v1/vehicles/{vehicle_id}/registry`. На стороне PostgreSQL используется мягкое удаление справочных полей транспорта: очищаются `monitoring_id`, `monitoring_name`, `monitoring_center`, `vehicle_full_name`, но сама строка `vehicles` не удаляется, чтобы не ломать связанные рейсы и историю.

## Создание рейса

Диалог `Создать рейс` собирает новую задачу в локальный payload и сам ничего не сохраняет. Сохранение выполняется через `NewTaskWorkflowService`.

Верхняя часть диалога:

- `ТС` — номер машины, с автодополнением по подстроке из `vehicles` и текущих строк задач;
- `Телефон` — ручной ввод, зарезервирован для будущей привязки к номеру ТС;
- `ФИО` — водитель, с автодополнением из `drivers` и текущих строк задач;
- `КА` — контрагент/перевозчик, с автодополнением из `carriers` и текущих строк задач;
- `ФИО2` — дополнительное поле водителя, с тем же автодополнением, пока не участвует в создании task.

Блок `Лентяй` содержит многострочное поле `Исходная заявка` высотой 120 px. Он нужен для быстрой вставки сырого текста заявки.

Поведение `Лентяй`:

- если вручную заполнена хотя бы одна точка `Погрузка` или `Выгрузка`, поле `Лентяй` не используется;
- если `Погрузка` и `Выгрузка` пустые, а `Лентяй` заполнен, текст делится по маркерам `Разгрузка` / `Выгрузка`;
- часть до маркера парсится как `Погрузка`, часть с маркера — как `Выгрузка`;
- парсинг выполняется через `Navigation_Bot.bots.route_info_parser.RouteInfoParser`;
- результат кладется в legacy-формат `buffer["Погрузка"]` / `buffer["Выгрузка"]`;
- сырые части сохраняются в `raw_load` / `raw_unload`;
- комментарии из блоков `Комментарий` сохраняются в `comm_load` / `comm_unload`.

Блоки `Погрузка` и `Выгрузка` встроены прямо в диалог. Они используют строки `AddressPointEditor`, как редактор адресов, и имеют кнопку `Добавить точку`. Минимальная высота каждого блока — 220 px, чтобы одна точка не схлопывала рабочую область, а несколько точек располагались стопкой.

## Зоны ответственности `CreateRaceDialog`

`CreateRaceDialog` остаётся UI-координатором сценария создания рейса. В нём должны оставаться:

- построение формы и кнопок;
- управление строками `AddressPointEditor`;
- базовая валидация обязательных полей;
- сбор `payload` через `get_payload()`.

Логика, которая не является разметкой диалога, вынесена в компоненты:

- `Navigation_Bot.gui.dialogs.components.create_race_suggestions.CreateRaceSuggestions` — собирает значения для autocomplete из `task_repository`, текущих строк и SQL-таблиц `vehicles`, `carriers`, `drivers`, затем устанавливает `QCompleter` на поля `ТС`, `КА`, `ФИО`, `ФИО2`;
- `Navigation_Bot.gui.dialogs.components.create_race_lazy_parser.CreateRaceLazyParser` — разбирает поле `Лентяй`, делит текст на погрузку/выгрузку и заполняет legacy-buffer через `RouteInfoParser`.

## Парсинг погрузок и выгрузок

Парсинг свободного текста вынесен из `DataCleaner` в `Navigation_Bot.bots.route_info_parser.RouteInfoParser`.

`DataCleaner` теперь отвечает за обработку строк репозитория:

- сохраняет исходные строки в `raw_load` / `raw_unload`;
- вызывает `RouteInfoParser.parse()` для полей `Погрузка` и `Выгрузка`;
- чистит поле `ТС`;
- привязывает Wialon ID через справочник vehicles;
- сохраняет результат через `TaskRepository`.

`RouteInfoParser` отвечает только за разбор текста маршрута:

- ищет точки по датам и нумерованным секциям;
- нормализует дату и время;
- отделяет адрес от служебных фраз, телефонов, контактов и комментариев;
- возвращает legacy-формат блоков `{ "Погрузка 1": ..., "Дата 1": ..., "Время 1": ... }` / `{ "Выгрузка 1": ... }`;
- добавляет общий блок `{ "Комментарий": ... }`, если в тексте найдены комментарии.

Важные эвристики:

- `RouteInfoParser` не считает фрагменты телефона временем, например `111-22-33` не должен давать `22:33`;
- нумерованные точки не распознаются внутри времени, например `16:00:00.` не должен становиться пунктом `00.`;
- короткие адресные сокращения `г.`, `ул.`, `с.`, `п.`, `д.` требуют точки, чтобы обычные слова вроде `СНЭК` не распознавались как начало адреса;
- если перед адресом идут контрагенты, например `ООО "ЛОРЕНЦ ...", ООО "Кюне-Нагель" Московская обл...`, контрагенты уходят в `Комментарий`, а адрес начинается с `Московская обл...`.

Регрессионные тесты парсера находятся в:

```powershell
python -m pytest tests\navigation_bot\bots\test_route_info_parser.py
```

При дальнейшем развитии диалога не стоит возвращать SQL-запросы, парсинг сырого текста или нормализацию payload обратно в `create_race_dialog.py`. Если поведение касается сохранения задачи, оно должно оставаться за `NewTaskWorkflowService`; если касается разбора текста заявки — за `CreateRaceLazyParser`; если касается источников подсказок — за `CreateRaceSuggestions`.

Для ручной проверки lazy-парсинга есть probe-тест:

```powershell
python -m pytest tests\navigation_bot\gui\test_create_race_lazy_parser.py -s
```

Текст заявки вставляется в файл:

```text
tests/navigation_bot/gui/lazy_text_probe_input.txt
```

Это обычный `.txt`, поэтому в него можно вставлять кавычки, `"""`, переносы и другие символы без экранирования Python-строк.

## Сценарий API

Клиентский репозиторий задач хранит активные фильтры:

- `include_completed`;
- `date_from`;
- `date_to`.

Эти значения передаются в `GET /api/v1/tasks` вместе с текущим `source_key`.

На стороне API параметры пробрасываются в `PostgresTaskReader.read_page()`, где:

- незавершенные задачи остаются режимом по умолчанию;
- завершенные, архивные и отмененные задачи включаются только при `include_completed=true`;
- диапазон дат применяется к первой доступной дате задачи из `planned_start_at`, `planned_end_at`, `completed_at`.

## Поведение обновления

Когда фильтры активны, инкрементальное обновление отключается и выполняется полная постраничная загрузка. Это нужно, чтобы результат не смешивал старые строки с новым набором фильтра.

Завершенные, архивные и отмененные задачи подсвечиваются отдельным спокойным зеленым фоном. Просроченные выгрузки для таких строк не подсвечиваются.
